# Спецификация: Справочник

## Дефолты

- **Имя** — PascalCase, существительное во **множественном** числе (`Сотрудники`, `Контрагенты`, `Товары`)
- **Видимость** — всегда `ВПодсистеме`. Менять на `ВПроекте` только если пользователь явно попросил
- **Иерархия** — не добавляй без явного запроса
- **Формы** — не указывай `Интерфейс.Объект.Форма` без явного запроса

## Иерархия

### Паттерн А — простая иерархия (`Иерархический: Истина`)

Платформа автоматически создаёт реквизит `Родитель` (тип `ИмяСправочника.Ссылка?`) и таблицы
`ИмяСправочника.Иерархия` / `ИмяСправочника.Иерархия.Циклы`.
**`Родитель` в `Реквизиты` НЕ добавлять — это системный реквизит.**
UUID: 1 (объект) + по 1 на каждый пользовательский реквизит.

```yaml
ВидЭлемента: Справочник
Ид: <UUID>
Имя: КатегорииТоваров
Иерархический: Истина
ОбластьВидимости: ВПодсистеме
Реквизиты:
    - Имя: Наименование
    - Ид: <UUID>
      Имя: Описание
      Тип: Строка
```

XBSL-запрос «все дочерние элементы, включая вложенные»:

```xbsl
знч Запрос = Запрос{
    ВЫБРАТЬ Ссылка ИЗ КатегорииТоваров
    ГДЕ Ссылка В ИЕРАРХИИ (%Родитель)
}
```

### Паттерн Б — дополнительные иерархии (`ДополнительныеИерархии`)

Когда родителем выступает другой справочник или нужно несколько иерархий.
Иерархическая связь строится по любому реквизиту-ссылке на другой справочник.
UUID: 1 (объект) + по 1 на каждый реквизит + **по 1 на каждую дополнительную иерархию**.

```yaml
ВидЭлемента: Справочник
Ид: <UUID>
Имя: Задачи
ОбластьВидимости: ВПодсистеме
ИерархияПоУмолчанию: Проект          # используется в В ИЕРАРХИИ без имени
ДополнительныеИерархии:
    -
        Ид: <UUID>
        Имя: Проект
        ПолеРодителя: Проект
    -
        Ид: <UUID>
        Имя: Исполнитель
        ПолеРодителя: Исполнитель
Реквизиты:
    - Имя: Наименование
    - Ид: <UUID>
      Имя: Проект
      Тип: Проекты.Ссылка?
    - Ид: <UUID>
      Имя: Исполнитель
      Тип: Сотрудники.Ссылка?
```

> При паттерне Б системного реквизита `Родитель` нет — связь через обычные реквизиты-ссылки.

## UUID

Нужны: 1 (объект) + по 1 на каждый пользовательский реквизит + по 1 на каждую табличную часть и каждый её реквизит.
Системные поля `Наименование` и `Код` — без UUID.

## Правила реквизитов

- `Наименование` — системное поле. Пишется как `- Имя: Наименование`, может добавляться `Длина` и `НезаполненноеЗначение: ЗапретитьВсегда`. Никаких других свойств (`Ид`, `Тип`, `МаксимальнаяДлина`).
- `Код` — системное поле. Не принимает `Ид` и `МаксимальнаяДлина`; принимает `Тип: Строка`, `Длина`, `Уникальность`, `СерииНумерации`, `Автонумерация`.
- Все остальные реквизиты требуют `Ид` (UUID).
- `НезаполненноеЗначение: ЗапретитьВсегда` — поле обязательно к заполнению (на `Наименование` и `Код` не ставится).
- `ИспользоватьВПолнотекстовомПоиске: Ложь` — для ссылочных полей, перечислений, бинарных данных.
- `ЗаполнятьПриКопировании: Ложь` — поле не копируется при дублировании записи.

## Межподсистемные ссылки

Когда реквизит ссылается на тип из **другой подсистемы Y**,
необходимо выполнить всю цепочку — иначе деплой завершится ошибками:

| Файл | Что добавить | Ошибка без этого |
|------|--------------|------------------|
| `Y/<Источник>.yaml` | `ОбластьВидимости: ВПроекте` | «Тип не виден из-за модификатора ВПодсистеме» |
| `X/Подсистема.yaml` | `Использование: [Y]` | «Подсистема не указана в качестве используемой» |
| `X/<НовыйСправочник>.yaml` | `Импорт: [Y]` | «Пространство имён не импортировано» |

Дефолт видимости — `ВПодсистеме`: объект доступен только внутри своей подсистемы.
Если он используется как тип реквизита из других подсистем — нужно явно указать `ОбластьВидимости: ВПроекте`.

Скилл `xbsl-meta-add` (Шаг 3а) выполняет эту проверку автоматически.
